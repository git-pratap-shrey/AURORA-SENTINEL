# Requirements Document: AI Scoring Improvements

## Introduction

This feature addresses critical accuracy issues in the AI-powered threat detection system. Currently, the system suffers from three major problems: (1) hardcoded fallback scores that cluster results around 40 regardless of actual threat level, (2) overly lenient threat classification that misidentifies real fights as "pranks" or "drama", and (3) an underutilized Nemotron ColEmbed V2 model that could provide enhanced verification but is never invoked in the analysis chain.

The improvements will eliminate hardcoded scores, implement stricter threat classification rules appropriate for public surveillance contexts, and integrate Nemotron as a secondary verification layer after Qwen2-VL analysis.

## Glossary

- **AI_Score**: A numerical threat assessment (0-100) generated by vision-language models analyzing video frames
- **ML_Score**: A numerical threat assessment (0-100) generated by machine learning models (YOLO pose detection, audio analysis)
- **Final_Score**: The weighted combination of ML_Score and AI_Score (30% ML + 70% AI)
- **Qwen2VL_Analyzer**: The primary GPU-accelerated vision-language model (Qwen2-VL-2B-Instruct) for threat analysis
- **Nemotron_Verifier**: The secondary verification model (nvidia/nemotron-colembed-vl-4b-v2) that uses embedding-based similarity scoring (NOT generative) to verify if Qwen2VL's text summary matches the actual image content
- **Scene_Type**: A classification category for detected activity (real_fight, organized_sport, normal, suspicious)
- **Threat_Classification_System**: The logic that determines Scene_Type based on AI analysis
- **Analysis_Summary**: A structured text description of Qwen2VL_Analyzer's findings passed to Nemotron_Verifier
- **Public_Surveillance_Context**: Video monitoring of public spaces where any physical aggression constitutes a threat
- **Organized_Sport**: Sanctioned combat activities with protective gear (boxing gloves, martial arts equipment, referee present)
- **Suspicious_Behavior**: Crowd patterns indicating concealment (people surrounding/covering two individuals) or unknown items in suspicious contexts

## Requirements

### Requirement 1: Eliminate Hardcoded AI Score Fallbacks

**User Story:** As a security operator, I want AI scores to accurately reflect the model's threat assessment, so that I can trust the system's risk evaluations instead of seeing artificial clustering around default values.

#### Acceptance Criteria

1. WHEN Qwen2VL_Analyzer fails to return a valid JSON response, THE Threat_Classification_System SHALL parse the text response using keyword analysis to extract a threat score
2. WHEN keyword analysis detects fight-related terms without sport indicators, THE Threat_Classification_System SHALL assign an AI_Score between 75-90 based on severity keywords
3. WHEN keyword analysis detects sport indicators with fight terms, THE Threat_Classification_System SHALL assign an AI_Score between 20-35
4. WHEN keyword analysis detects normal activity indicators, THE Threat_Classification_System SHALL assign an AI_Score between 10-25
5. THE Threat_Classification_System SHALL NOT assign a fixed fallback score of 40 under any circumstances
6. WHEN all AI models fail to provide analysis, THE Threat_Classification_System SHALL use ML_Score as the Final_Score with a confidence indicator of 0.3
7. FOR ALL AI_Score calculations, the score SHALL be derived from either parsed JSON, keyword analysis, or ML_Score passthrough (no hardcoded defaults)

### Requirement 2: Implement Strict Public Surveillance Threat Classification

**User Story:** As a security operator monitoring public spaces, I want the system to classify any physical aggression as a threat, so that real fights are not dismissed as "pranks" or "drama" and I can respond appropriately.

#### Acceptance Criteria

1. THE Threat_Classification_System SHALL remove "prank" and "drama" from the valid Scene_Type values in Public_Surveillance_Context
2. THE Threat_Classification_System SHALL support exactly three Scene_Type values: "real_fight", "organized_sport", "normal", "suspicious"
3. WHEN Qwen2VL_Analyzer detects physical aggression in Public_Surveillance_Context, THE Threat_Classification_System SHALL classify Scene_Type as "real_fight" unless Organized_Sport indicators are present
4. WHEN Qwen2VL_Analyzer detects protective gear (boxing gloves, martial arts equipment) AND a referee or ring structure, THE Threat_Classification_System SHALL classify Scene_Type as "organized_sport"
5. WHEN Scene_Type is "organized_sport", THE Threat_Classification_System SHALL cap AI_Score at 35
6. WHEN Qwen2VL_Analyzer detects heavy fighting (multiple strikes, sustained aggression, visible injury), THE Threat_Classification_System SHALL assign AI_Score between 80-95
7. WHEN Qwen2VL_Analyzer detects Suspicious_Behavior (crowd surrounding two people, unknown items in suspicious contexts), THE Threat_Classification_System SHALL classify Scene_Type as "suspicious" and assign AI_Score between 60-75
8. THE Threat_Classification_System SHALL update all analysis prompts to remove references to "prank" and "drama" classifications
9. FOR ALL threat classifications in Public_Surveillance_Context, any physical contact with aggressive intent SHALL be classified as "real_fight" unless explicitly identified as Organized_Sport

### Requirement 3: Integrate Nemotron as Secondary Verification Layer

**User Story:** As a system architect, I want Nemotron to verify that Qwen2-VL's text summary accurately describes the image content using embedding-based similarity scoring, so that we catch misclassifications and improve threat detection accuracy.

**CRITICAL NOTE:** Nemotron ColEmbed V2 is an embedding model, NOT a generative/chat model. It uses `forward_queries()`, `forward_images()`, and `get_scores()` methods to compute similarity between text and images. It CANNOT generate summaries or chat responses.

#### Acceptance Criteria

1. WHEN Qwen2VL_Analyzer completes its analysis, THE Threat_Classification_System SHALL extract the text explanation/summary from the response
2. THE Nemotron_Verifier SHALL compute embeddings for: (a) the original frame image using `forward_images()`, (b) Qwen2VL's text summary using `forward_queries()`, (c) predefined threat category queries using `forward_queries()`
3. THE predefined threat queries SHALL include: "real fight with violence and aggression in public space", "organized sport with protective gear and referee", "normal safe activity with no threats", "suspicious crowd behavior covering people or unknown items"
4. THE Nemotron_Verifier SHALL compute similarity scores using `get_scores()` between: (a) image ↔ Qwen2VL summary (verification score), (b) image ↔ each threat category (classification scores)
5. WHEN the image ↔ Qwen2VL summary similarity is above 0.6, THE Nemotron_Verifier SHALL mark the analysis as "verified" (Qwen2VL's description matches the image)
6. WHEN the image ↔ Qwen2VL summary similarity is below 0.6, THE Nemotron_Verifier SHALL mark the analysis as "mismatch" (Qwen2VL may have misunderstood the image)
7. THE Nemotron_Verifier SHALL identify the highest-scoring threat category from the image ↔ threat query similarities
8. WHEN Nemotron's highest threat category matches Qwen2VL's Scene_Type AND verification score > 0.6, THE Threat_Classification_System SHALL use the average of both AI_Scores with high confidence
9. WHEN Nemotron's highest threat category contradicts Qwen2VL's Scene_Type OR verification score < 0.6, THE Threat_Classification_System SHALL flag for review and use the higher risk score (conservative approach)
10. THE Nemotron_Verifier SHALL map threat category similarities to risk scores: >0.7 to "real fight" = 80-95, >0.7 to "organized sport" = 20-35, >0.7 to "normal" = 10-25, >0.7 to "suspicious" = 60-75
11. WHEN Nemotron_Verifier fails or is unavailable, THE Threat_Classification_System SHALL use Qwen2VL_Analyzer's AI_Score without modification and log the fallback
12. THE Threat_Classification_System SHALL include both Qwen2VL_Analyzer and Nemotron_Verifier outputs (verification score, category scores, agreement status) in the final analysis result for audit purposes
13. THE Nemotron_Verifier SHALL complete all embedding computations and scoring within 3 seconds to maintain system performance requirements

### Requirement 4: Update Weighted Scoring Logic

**User Story:** As a security operator, I want the final threat score to properly reflect both ML and AI assessments, so that the system provides accurate risk evaluations even when AI models disagree or fail.

#### Acceptance Criteria

1. WHEN both ML_Score and AI_Score are available, THE Threat_Classification_System SHALL calculate Final_Score as (0.3 × ML_Score + 0.7 × AI_Score)
2. WHEN AI_Score is unavailable due to model failure, THE Threat_Classification_System SHALL set Final_Score equal to ML_Score and set confidence to 0.3
3. WHEN ML_Score is unavailable, THE Threat_Classification_System SHALL set Final_Score equal to AI_Score and set confidence to 0.6
4. THE Threat_Classification_System SHALL NOT apply arbitrary multipliers (such as 0.6 × ML_Score) as fallback scores
5. THE Threat_Classification_System SHALL include metadata in the response indicating which scores were used (ml_only, ai_only, or weighted)
6. WHEN Nemotron_Verifier provides an adjusted score, THE Threat_Classification_System SHALL use the Nemotron-adjusted AI_Score in the weighted calculation
7. FOR ALL scoring calculations, the system SHALL log the component scores (ML_Score, AI_Score, Nemotron adjustment) for debugging and audit purposes

### Requirement 5: Enhance Analysis Prompts for Strict Classification

**User Story:** As a system architect, I want AI model prompts to guide toward strict threat classification, so that models are less likely to misclassify real threats as benign activities.

#### Acceptance Criteria

1. THE Threat_Classification_System SHALL use prompts that explicitly state the Public_Surveillance_Context
2. THE Qwen2VL_Analyzer prompts SHALL instruct the model to differentiate only between: real threats, Organized_Sport, normal activity, and Suspicious_Behavior
3. THE Qwen2VL_Analyzer prompts SHALL include specific indicators for Organized_Sport: protective gear, referee presence, ring/mat structure
4. THE Qwen2VL_Analyzer prompts SHALL instruct that any fight without Organized_Sport indicators is a real threat
5. THE Nemotron_Verifier prompts SHALL ask the model to verify the Qwen2VL_Analyzer's classification and identify any missed threat indicators
6. THE Nemotron_Verifier prompts SHALL include the Analysis_Summary from Qwen2VL_Analyzer as context
7. THE Threat_Classification_System SHALL remove all prompt text mentioning "prank", "drama", "staged", or "performance" classifications
8. FOR ALL prompts, the system SHALL emphasize that heavy fighting (multiple strikes, sustained aggression) indicates high threat regardless of context

### Requirement 6: Implement Robust Error Handling

**User Story:** As a system operator, I want the system to gracefully handle AI model failures, so that threat detection continues functioning even when individual models are unavailable.

#### Acceptance Criteria

1. WHEN Qwen2VL_Analyzer fails to load, THE Threat_Classification_System SHALL log the error and attempt to use Ollama as fallback
2. WHEN Nemotron_Verifier fails to load at initialization, THE Threat_Classification_System SHALL log the error and continue with single-model analysis
3. WHEN Nemotron_Verifier times out during analysis, THE Threat_Classification_System SHALL use Qwen2VL_Analyzer's score and log the timeout
4. WHEN both Qwen2VL_Analyzer and Ollama fail, THE Threat_Classification_System SHALL use ML_Score as Final_Score
5. THE Threat_Classification_System SHALL include error details in the analysis response for debugging
6. THE Threat_Classification_System SHALL track model availability status and report it in health check endpoints
7. WHEN a model fails three consecutive times, THE Threat_Classification_System SHALL mark it as unavailable and skip it for 5 minutes before retrying
8. FOR ALL error conditions, the system SHALL ensure that some threat score is returned (never null or undefined)

### Requirement 7: Maintain Performance Requirements

**User Story:** As a system operator, I want AI analysis to complete quickly, so that real-time threat detection remains responsive for live video feeds.

#### Acceptance Criteria

1. WHEN analyzing a single frame, THE Qwen2VL_Analyzer SHALL complete analysis within 2 seconds on GPU-enabled systems
2. WHEN Nemotron_Verifier is invoked, THE verification SHALL complete within 3 seconds
3. THE combined Qwen2VL_Analyzer and Nemotron_Verifier analysis SHALL complete within 5 seconds total
4. WHEN analysis exceeds time limits, THE Threat_Classification_System SHALL timeout and use available partial results
5. THE Threat_Classification_System SHALL log analysis latency for each model in the response metadata
6. WHEN ML_Score is below 20, THE Threat_Classification_System SHALL skip AI analysis to conserve resources
7. THE Threat_Classification_System SHALL support concurrent analysis of multiple frames without blocking
8. FOR ALL analysis operations, memory usage SHALL remain within 6GB RAM limits on the target hardware (RTX 4050, 6GB RAM)
